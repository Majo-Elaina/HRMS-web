# HRMS AI Agent 执行方案

## 目标

在当前 HRMS 基础功能稳定后，引入一个“先规划、后审批、再执行”的 AI agent。用户用自然语言描述需求，agent 先生成操作步骤、影响范围、接口调用和关键校验项，人工检查并明确同意后，agent 才执行系统操作。

## 核心原则

- AI 未获批准前不得执行写操作。
- AI 只能调用现有后端 API，不能绕过权限直接写库。
- AI 权限不能高于当前登录用户。
- 所有执行必须有计划、审批、日志和结果回执。
- 高风险操作必须支持前置校验，优先支持 dry-run。

## 推荐架构

### 前端

增加“AI 助手”入口，负责：

- 输入自然语言需求
- 展示结构化执行计划
- 让人工批准或拒绝执行
- 展示执行进度和执行日志

### Agent 编排层

建议独立成 `agent-service`，也可以先放在当前后端项目中。

拆成四个能力：

- `planner`：需求解析和步骤规划
- `validator`：权限、参数、目标对象校验
- `executor`：按步骤调用业务 API
- `auditor`：记录审批、执行、异常和回滚日志

### 业务 API 层

agent 只允许调用白名单接口，例如：

- 员工、部门、职位、角色、权限
- 考勤、请假、薪资
- 审批规则、模块范围

建议补充：

- 幂等键 `idempotencyKey`
- 操作日志
- 前置校验接口
- `dryRun` 支持

### 审计数据层

建议新增：

- `agent_task`
- `agent_task_step`
- `agent_execution_log`
- `agent_approval_record`

## 典型流程

### 1. 用户提需求

例如：

- “为财务专员增加薪资记录查看权限”
- “把张伟 3 月 12 日的考勤补录为正常签到”
- “帮我提交 2 天病假申请”

### 2. agent 生成计划

计划至少包含：

- 需求理解
- 涉及模块
- 目标对象
- 执行步骤
- 每一步调用的 API、方法和参数摘要
- 风险说明
- 回滚建议

### 3. 人工审核

人工确认：

- 目标对象是否正确
- 影响范围是否合理
- 是否涉及批量变更
- 是否会触发连带更新

### 4. 批准后执行

执行策略建议：

- 严格串行
- 每一步先预检查
- 失败即停
- 记录完整执行日志

### 5. 返回回执

执行结果应包含：

- 成功/失败
- 已完成步骤
- 失败原因
- 实际写入结果摘要
- 是否回滚

## 计划 JSON 建议

```json
{
  "intent": "update_role_permission",
  "summary": "为财务专员增加薪资记录查看权限",
  "riskLevel": "medium",
  "requiresApproval": true,
  "entities": [
    { "type": "role", "id": 6, "name": "财务专员" }
  ],
  "steps": [
    {
      "stepNo": 1,
      "title": "查询角色当前权限",
      "method": "GET",
      "api": "/api/role-permissions/role/6/perm-ids"
    },
    {
      "stepNo": 2,
      "title": "合并权限并提交保存",
      "method": "PUT",
      "api": "/api/role-permissions/role/6",
      "payloadPreview": {
        "permIds": [1, 2, 3, 25]
      }
    }
  ],
  "rollbackPlan": [
    "保存旧权限集",
    "失败时按旧权限集恢复"
  ]
}
```

## 适合优先接入的场景

建议从低风险、强结构化场景开始：

1. 查询员工、考勤、请假、薪资摘要
2. 提交个人请假申请
3. 补录单条考勤
4. 修改单个员工基础信息
5. 调整单个角色的权限项

暂不建议一开始就放开：

- 批量删除
- 批量改薪资
- 批量调岗
- 跨模块的大规模联动修改

## 后端需要补的能力

### 统一接口白名单

对每个可调用接口标记：

- 功能名称
- 风险等级
- 是否只读
- 必填参数
- 是否支持回滚

### Dry-run

高风险写操作先做预检查，例如：

- 目标记录是否存在
- 当前用户是否有权限
- 本次变更会影响哪些记录

### 审计日志

必须记录：

- 谁提的需求
- agent 如何理解
- 谁批准的
- 实际调用了哪些接口
- 请求参数摘要和返回摘要

### 回滚策略

常见场景至少准备补偿方案：

- 角色权限：恢复旧权限集
- 员工信息：恢复修改前快照
- 请假状态：恢复旧状态和审批链

## 前端交互建议

建议做成三段式：

### 需求输入区

- 文本输入框
- 常用示例
- “仅生成方案”开关

### 计划展示区

- 需求理解
- 影响对象
- 步骤清单
- 风险等级
- 回滚方案

### 审批执行区

- `生成计划`
- `重新规划`
- `批准并执行`
- `拒绝执行`
- `查看执行日志`

## 大模型接入建议

采用“两段式”：

### 规划模型

输入：

- 用户需求
- 当前用户身份
- 系统模块说明
- API 白名单
- 固定 JSON 输出约束

输出：

- 结构化执行计划

### 执行代理

执行代理不自由发挥，只消费规划结果 JSON：

- 做参数补全
- 做前置校验
- 按步骤调用接口
- 记录日志

## 在当前项目里的最小落地顺序

1. 整理 agent 可调用的接口白名单
2. 新增 `agent_task / agent_task_step / agent_execution_log / agent_approval_record`
3. 新增“生成执行计划”接口
4. 前端先做“只生成方案，不执行”
5. 再增加“批准后执行”
6. 第一批只接请假申请、单条考勤补录、角色权限调整

## 结论

要实现“告诉 agent 需求，agent 先写步骤，经同意后再执行”，关键是四层：

- 结构化规划
- 人工审批门禁
- 受控 API 执行
- 全量审计和回滚

这四层做稳后，AI 才能成为系统里的受控操作代理，而不是不可追踪的自动脚本。
