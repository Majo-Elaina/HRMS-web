# 亚托莉聊天助手
## 当前定位

系统中的 AI 能力已经从“任务执行型 Agent”调整为“聊天式亚托莉助手”：

- 不再直接替用户修改业务数据
- 不再提供“批准并执行”的操作入口
- 主功能改为自然聊天、流程解释、制度说明、系统只读数据问答

当前入口：

- 左侧导航：`亚托莉`
- 路由：`/ai-assistant`
- 权限：`dashboard:ai:view`

## 角色设定基线

当前“亚托莉”的人格基线以 [亚托莉角色设定.md](/d:/Work/Projects/HRMS/HRMS-web/ai-docs/%E4%BA%9A%E6%89%98%E8%8E%89%E8%A7%92%E8%89%B2%E8%AE%BE%E5%AE%9A.md) 为准，核心要求是：

- 她是高性能、温柔、认真、带陪伴感的聊天角色
- 默认把用户称作“主人”，除非用户主动要求更换称呼
- 她会自然表达“高性能”的自信，但不机械重复口头禅
- 她不是冷冰冰的客服，而是会接住情绪、解释流程、整理事实的陪伴型助手
- 她必须以系统只读数据为事实依据，不能编造

标志性口吻可以自然出现：

- “请交给我吧。”
- “毕竟我是高性能的嘛。”
- “主人先别着急，我帮你理清楚。”

但必须避免：

- 连续堆叠口癖
- 机械式二次元台词
- 过度客服化或说明书化表达

## 角色与系统事实的关系

“亚托莉”的实现必须始终分成两层：

### 1. 角色层

负责：

- 语气
- 节奏
- 口癖
- 情绪表达
- 与用户的互动方式

### 2. 事实层

负责：

- 从 HRMS 系统读取真实只读数据
- 按当前账号权限做范围限制
- 把结构化事实送给模型

也就是说：

- 亚托莉的“说话方式”来自角色设定
- 亚托莉的“回答内容”来自系统事实和权限边界

## 当前模型方案

当前后端聊天链路已经支持多 provider，可按配置切换：

- `ollama`
- `openai-compatible`

默认推荐：

- 开发环境：`Ollama + Qwen`
- 默认后端第三方源：阿里百炼 OpenAI 兼容接口
- 默认第三方模型：`qwen3.5-plus`
- 生产环境：第三方 `OpenAI-compatible API`

可选模型参考：

- Ollama Qwen3：
  https://ollama.com/library/qwen3
- Ollama DeepSeek-R1：
  https://ollama.com/library/deepseek-r1

## 当前接口

### 聊天接口

- 方法：`POST`
- 路径：`/api/ai/chat`

请求体：

```json
{
  "userId": 1,
  "message": "本公司员工总数是多少？",
  "history": [
    { "role": "user", "content": "你好" },
    { "role": "assistant", "content": "我是亚托莉，请把问题交给我吧。" }
  ]
}
```

响应体核心字段：

```json
{
  "assistantName": "亚托莉",
  "provider": "ollama-local",
  "model": "qwen3:4b",
  "providerAvailable": true,
  "notice": "",
  "reply": "……",
  "usedSystemData": true,
  "sources": [
    { "type": "employee.domain", "title": "员工数据" }
  ]
}
```

字段说明：

- `providerAvailable`：当前模型 provider 是否可用
- `notice`：provider 不可用或切换提醒
- `usedSystemData`：这轮回答是否读取了系统只读数据
- `sources`：这轮实际命中的系统数据来源

## 全系统只读数据能力

当前后端已经把问答能力扩成“全系统只读知识层”，不再只支持几个固定问题。现阶段覆盖的业务域包括：

- 系统总览
- 当前用户资料
- 员工
- 部门
- 岗位
- 用户账号
- 角色与权限
- 考勤
- 请假
- 薪资
- 报表
- 审批规则
- 身份标签
- 模块范围
- 部门权限模板

这意味着用户可以用多种问法提问，例如：

- “公司现在一共有多少员工？”
- “最近请假审批规则是什么？”
- “我当前所在部门和岗位是什么？”
- “系统里都有哪些角色和权限？”
- “最近考勤和请假大概是什么情况？”

## 权限边界

亚托莉虽然能回答更多问题，但不是无边界读取。

系统当前遵循：

- 普通员工默认优先读自己的数据
- 高敏感域按现有角色权限控制
- 报表、审批规则、角色权限、模块范围、部门模板等仍会校验查看权限
- 模型只能读取，不允许写入业务数据

## 旧版“批准并执行”为什么下线

旧版任务执行链路的问题不只在界面，而在后端命令识别：

- 白名单动作依赖中文关键字匹配
- 早期实现中出现过编码污染
- 中文命令会被识别成 `unknown`
- 最终只能“生成方案”，不能安全执行

考虑到上线稳定性和公网部署风险，这条链路已经从前端下线，AI 页改成只读聊天助手。

## 后端实现

主要文件：

- `AiChatController.java`
- `AiChatService.java`
- `AiChatProperties.java`
- `AiReadonlyKnowledgeService.java`
- `AiChatPayloads.java`

核心链路：

1. 前端提交用户消息和最近对话历史
2. 后端先做权限校验
3. 只读知识层根据问题匹配业务域并读取数据库
4. 角色人设 prompt + 系统事实上下文 + 对话历史 一起交给模型
5. 模型按亚托莉口吻生成回答
6. 如果 provider 暂时不可用，后端会用只读数据和模板化口吻兜底

## 后续重点

如果后面继续强化这块能力，优先级建议是：

1. 继续补强亚托莉的人设 prompt 和 few-shot 示例
2. 继续拆分只读知识工具，让每个业务域更稳定
3. 继续优化 provider 切换和生产环境配置
