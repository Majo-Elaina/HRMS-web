# 企业人事管理系统总览

## 1. 系统定位

本系统本质上是一个中小企业内部的人事 + OA + 薪资 + 权限协同系统，核心目标是把以下链路打通：

- 组织管理：部门、职位、员工、系统用户
- 权限管理：角色权限、身份标签、部门模板、模块数据范围
- OA 审批：请假审批、薪资记录审批、薪资配置审批
- 业务执行：考勤记录、请假申请、薪资发放
- 经营看板：首页 Dashboard、统计报表 Report

## 2. 整体架构

### 2.1 技术架构

系统采用典型的前后端分离三层结构：

1. 前端展示层
   - 技术栈：`Vue 3 + Vite + Pinia + Vue Router + Element Plus + Axios + ECharts`
   - 主要职责：
     - 登录态维护
     - 路由守卫与菜单权限控制
     - 页面交互、表单录入、审批操作
     - 报表与看板可视化

2. 后端业务层
   - 技术栈：`Spring Boot + Spring Data JPA`
   - 主要职责：
     - 用户认证与资料组装
     - 组织、员工、用户、角色、权限 CRUD
     - 审批规则、模块范围、身份标签等配置管理
     - 请假 / 考勤 / 薪资业务持久化
     - 汇总报表接口输出

3. 数据持久层
   - 数据库：`MySQL`
   - ORM：`JPA/Hibernate`
   - 主要职责：
     - 结构化存储员工、审批、薪资、权限等核心数据
     - 用外键保证角色、权限、部门、员工、审批规则之间的关系一致性

### 2.3 权限控制模型

系统的权限控制不是单一模型，而是四层叠加：

1. 角色权限
   - 通过 `role_permission` 决定用户能不能访问某个菜单/按钮
   - 例如：`attendance:leave:approve`、`salary:record:pay`

2. 身份标签
   - 通过 `employee.identity_tag_code` 标记员工业务身份
   - 例如：`HR_SPECIALIST`、`MANAGER`、`FINANCE_MANAGER`
   - 主要影响审批资格和模块数据范围

3. 模块数据范围
   - 通过 `module_scope_rule + module_scope_detail` 决定一个标签在某模块能看“本人/本部门/全公司”数据
   - 例如普通员工只能看自己的考勤，HR 可以看公司范围员工数据

4. 审批规则
   - 通过 `approval_rule_type + approval_rule` 决定不同业务单据如何流转审批人
   - 例如请假按天数和申请人标签走不同审批链

## 3. OA / 审批逻辑

### 3.1 审批规则模型

审批规则分两层：

- 审批类型：`approval_rule_type`
  - `leave`
  - `salary_record`
  - `salary_config`
  - 后续也可扩展新的业务类型

- 审批规则：`approval_rule`
  - `applicantTag`：申请人身份标签
  - `daysOp` + `daysValue`：针对请假场景的天数条件
  - `firstApproverTag`：一级审批人标签
  - `secondApproverTag`：二级审批人标签
  - `secondApproverScope`：二级审批范围（部门/公司等）
  - `sortOrder`：命中顺序

前端在请假页和首页待办里，会基于“申请人标签 + 天数”匹配命中的审批规则。

### 3.2 请假审批逻辑

请假是本系统最完整的 OA 链路，逻辑如下：

1. 员工发起请假申请
   - 新建 `leave_request`
   - 系统根据申请人身份标签与请假天数命中一条审批规则

2. 写入一级待审批人
   - `pendingApproverTag = firstApproverTag`
   - `pendingApproverScope = company/dept/self`

3. 审批人处理
   - 当前登录人必须同时满足：
     - 拥有 `attendance:leave:approve`
     - 身份标签匹配 `pendingApproverTag`
     - 数据范围匹配 `pendingApproverScope`

4. 若存在二级审批
   - 一级审批通过后，把：
     - `pendingApproverTag = nextApproverTag`
     - `pendingApproverScope = nextApproverScope`
   - 单据继续流转

5. 终审结果
   - 全部通过：单据进入通过状态
   - 任意一环拒绝：单据进入拒绝状态
   - 申请人撤销：单据进入撤销状态

### 3.3 薪资记录审批逻辑

薪资记录的 OA 逻辑相对固定：

1. 财务专员录入或维护薪资记录
2. 具备 `salary:record:submit` 权限的角色提交审批
3. 财务经理或具备 `salary:record:approve` 权限的角色审批
4. 具备 `salary:record:pay` 权限的角色执行发放

在当前系统中，这条链路通常对应：

- 财务专员：录入、编辑、提交
- 财务经理：审批、发放

### 3.4 薪资配置审批逻辑

薪资配置用于维护工资计算规则或参数，逻辑如下：

1. 财务专员新增或修改配置
2. 具备 `salary:config:submit` 权限的角色提交
3. 财务经理或具备 `salary:config:approve` 权限的角色审批生效

这条链路保证配置项的修改不会绕过审批直接生效。

### 3.5 模块范围与 OA 的协同逻辑

模块数据范围会直接影响审批人与业务视角：

- 普通员工
  - 通常只能看“自己”的请假、考勤、工资
- 部门经理
  - 通常可看“本部门”的人员与审批待办
- HR / 财务
  - 通常可看“全公司”或更广范围的数据

因此，本系统的 OA 不是单纯“谁有按钮谁就能批”，而是：

- 权限决定“能不能操作”
- 身份标签决定“是不是这个岗位角色”
- 模块范围决定“能不能看到这条单据”

## 4. 职位 / 角色 / 功能矩阵

说明：

- “角色”是系统权限角色，来源于 `role`
- “职位/身份”是业务岗位语义，主要通过 `identityTag` 表达
- 实际权限以数据库 `role_permission` 为准；下表是当前代码与默认规则表达出的业务意图

### 4.1 系统管理员 `ADMIN`

核心职责：

- 系统初始化与全局治理
- 角色、权限、规则、模板的最高管理者

典型功能：

- 首页总览
- 员工 / 部门 / 职位全部管理
- 用户管理
- 角色管理与角色授权
- 身份标签管理
- 部门权限模板管理
- 模块范围配置
- 审批规则配置
- 报表查看

权限特点：

- 拥有全部权限码
- 数据范围通常为全公司

### 4.2 HR 专员 `HR`

核心职责：

- 人事基础资料维护
- 请假流程中的人事审批参与者

典型功能：

- 员工信息管理
- 部门管理
- 职位管理
- 考勤记录维护
- 请假申请查看与审批
- 基础报表查看

权限特点：

- 有人事基础信息增删改
- 有请假审批权限
- 无角色授权、无系统级配置能力

### 4.3 HR 经理 `HR_MANAGER`

核心职责：

- 人事业务负责人
- 请假审批高级节点

典型功能：

- HR 专员全部功能
- 更高层级的人事审批与规则确认

权限特点：

- 与 HR 专员接近，但在 OA 审批链条中层级更高

### 4.4 部门经理 `MANAGER`

核心职责：

- 管理本部门员工
- 处理本部门相关审批

典型功能：

- 查看/编辑本部门员工信息
- 查看考勤
- 审批请假
- 查看本部门报表

权限特点：

- 更偏“部门管理者”视角
- 数据范围通常限制在本部门

### 4.5 普通员工 `EMPLOYEE`

核心职责：

- 员工自助

典型功能：

- 查看个人首页信息
- 查看个人考勤记录
- 发起请假申请
- 撤销自己的请假申请
- 查看个人薪资记录

权限特点：

- 不具备组织管理能力
- 数据范围通常为本人

### 4.6 财务专员 `FINANCE`

业务身份通常映射为 `FINANCE_SPECIALIST`

核心职责：

- 薪资数据执行层

典型功能：

- 新增 / 编辑薪资记录
- 提交薪资记录审批
- 新增 / 编辑薪资配置
- 提交薪资配置审批

权限特点：

- 负责“制作与提交”
- 不负责最终批准和发放

### 4.7 财务经理 `FINANCE_MANAGER`

业务身份通常映射为 `FINANCE_MANAGER`

核心职责：

- 薪资审批与最终发放负责人

典型功能：

- 审批薪资记录
- 发放工资
- 审批薪资配置

权限特点：

- 负责“审批与执行”
- 对薪资模块拥有更高权限

## 5. 主要业务链路

### 5.1 登录与权限装载

1. 用户通过 `/api/auth/login` 登录
2. 后端按 `sys_user -> employee -> role -> role_permission` 组装 `UserProfile`
3. 前端把 `permissions`、`identityTag`、`deptId` 等写入用户上下文
4. 前端继续加载：
   - 模块范围配置
   - 审批规则类型
   - 审批规则
5. 路由守卫按 `meta.permission` 判断能否进入页面

### 5.2 菜单与按钮控制

控制顺序如下：

1. 路由级权限
   - 例如页面要求 `base:employee:view`

2. 按钮级权限
   - 例如新增按钮要求 `base:employee:add`

3. 数据范围控制
   - 即使能进页面，也未必能看到全量数据

### 5.3 仪表盘与报表

Dashboard / Report 不是单独存表，而是组合多个接口形成：

- 员工
- 部门
- 职位
- 考勤
- 请假
- 薪资
- 审批规则
- 汇总接口 `/api/report/summary`

因此它们属于“聚合展示层”，不是独立业务源。

## 6. 当前系统边界与设计特点

### 6.1 已具备的能力

- 完整的人事基础信息管理
- 基于角色的菜单/按钮权限控制
- 基于身份标签和范围规则的数据权限控制
- 请假 / 薪资配置 / 薪资记录的审批链条
- 统计看板与报表展示

### 6.2 当前设计特点

- 角色权限负责“能做什么”
- 身份标签负责“你是谁”
- 模块范围负责“你能看到谁的数据”
- 审批规则负责“单据怎么流转”

这四者是并行协同，不是彼此替代。

### 6.3 适合继续扩展的方向

- 补充更多 OA 类型
  - 出差审批
  - 采购审批
  - 加班审批
- 增加操作日志查询与审计页面
- 增加审批流可视化
- 增加消息通知 / 待办提醒

## 7. 一句话总结

这个系统的本质是：

> 用“角色权限 + 身份标签 + 数据范围 + 审批规则”四层模型，把企业内部的人事、薪资和 OA 审批统一在一套可配置、可持久化的管理平台里。
