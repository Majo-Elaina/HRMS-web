# 权限与审批模型优化方案

## 1. 本次问题的直接结论

当前“财务专员提交请假后，申请人侧显示待 HR 专员审批，但 HR 账号侧却表现异常”的根因，不是单独因为“前端删除了身份标签功能”，而是系统内同时存在三套口径：

1. 审批规则匹配，按 `identity_tag_code` 这类业务身份标签判断。
2. 审批资格校验，很多地方却按 `roleCode` 精确相等判断。
3. 状态展示文案，又混用了“待审批 / 待一级审批 / 待 HR 专员审批”三种表达。

这会导致：

- 申请单匹配出来的一级审批人是 `HR_SPECIALIST`
- 登录用户真实角色编码却是 `HR`
- 页面判断“当前人能不能审批”时因为 `HR_SPECIALIST !== HR` 失败
- 同一张单在不同页面出现不同状态文案或不同待办结果

所以，问题本质是“身份标签、角色、审批节点字段的语义边界没有完全收口”，不是单一功能删掉造成的。

## 2. 现状诊断

### 2.1 四层框架本身是合理的

现有四层大框架应当保留：

1. 身份标签：表达业务身份是谁
2. 角色：表达系统功能能做什么
3. 模块范围：表达数据能看到谁
4. 审批规则：表达单据怎么流转

问题不在框架层次，而在每层之间存在交叉代用。

### 2.2 当前最核心的混乱点

#### A. “审批人字段”名字叫 Tag，页面实际当 Role 用

`approval_rule.first_approver_tag`、`leave_request.pending_approver_tag` 从命名上看是“身份标签”，但审批规则页面里审批人下拉来源是角色列表，保存的是 `roleCode`。

结果就是：

- 申请人维度，使用的是身份标签体系：`EMPLOYEE / HR_SPECIALIST / FINANCE_SPECIALIST`
- 审批人维度，有时保存身份标签，有时保存角色编码：`HR / HR_MANAGER / FINANCE_MANAGER`

这会让“谁来批”变成一个不稳定概念。

#### B. 特殊岗位既被当普通员工，又需要特殊权限

现在 HR 专员、财务专员在审批规则和模块范围里被当成特殊业务身份；
但在某些前端逻辑里又被视作普通员工别名，例如：

- `HR_SPECIALIST -> EMPLOYEE`
- `FINANCE_SPECIALIST -> EMPLOYEE`

这个映射适合做“兜底匹配”，不适合做“主判断”。

#### C. 审批状态被前端二次推导

当前请假单的状态流转与审批链修正，仍主要在前端完成。这样会产生两个问题：

1. 任意页面都可能各自推导一套审批链
2. 历史数据会被页面加载时“顺手改写”

这类逻辑一旦散在多个页面，后续必然继续分叉。

## 3. 优化目标

在不改变“四层框架”前提下，把每层职责彻底收口：

1. 身份标签：只回答“他是谁”
2. 角色：只回答“他能做什么功能”
3. 模块范围：只回答“他能看哪些数据”
4. 审批规则：只回答“单据下一步交给哪一类审批人”

## 4. 推荐的语义收口方案

### 4.1 身份标签层

身份标签保留，并升级为“业务主身份”。

建议标签集合：

- `ADMIN`
- `GENERAL_MANAGER`
- `HR_MANAGER`
- `HR_SPECIALIST`
- `FINANCE_MANAGER`
- `FINANCE_SPECIALIST`
- `MANAGER`
- `EMPLOYEE`

规则：

1. 每个员工必须且只能有一个主身份标签。
2. `employee.identity_tag_code` 是审批匹配、模块范围匹配的唯一业务身份来源。
3. 前端推导身份标签只允许作为兼容兜底，不允许作为正式主逻辑。

### 4.2 角色层

角色只保留系统功能权限职责，不再承担业务身份推理。

建议角色定位：

- `ADMIN`：系统治理
- `GENERAL_MANAGER`：公司级查看与高层审批
- `HR`：人事功能操作角色
- `HR_MANAGER`：人事高级管理角色
- `FINANCE`：财务操作角色
- `FINANCE_MANAGER`：财务审批与发放角色
- `MANAGER`：部门管理角色
- `EMPLOYEE`：员工自助角色

规则：

1. 角色决定菜单、按钮、API 功能权限。
2. 角色不直接参与审批规则命中。
3. 角色与身份标签允许不是同一编码，但必须有稳定映射关系。

### 4.3 模块范围层

模块范围只按身份标签配置，不按角色配置。

原因：

- 模块范围本质上是业务视角，不是菜单视角
- HR 专员和 HR 经理、财务专员和财务经理在数据可见范围上天然更适合按身份标签配置

建议：

1. `module_scope_detail.tag_code` 继续只存身份标签。
2. 页面获取数据范围时统一调用“当前用户主身份标签 -> 模块范围”。
3. 页面不再根据角色去猜 scope。

### 4.4 审批规则层

审批规则继续保留，但要把“申请人”和“审批人”彻底拆清。

建议保留现表结构，但统一语义为：

- `applicantTag`：申请人身份标签
- `firstApproverTag`：一级审批岗位标识
- `secondApproverTag`：二级审批岗位标识

这里的关键建议是：

1. 申请人始终使用身份标签。
2. 审批人也统一使用“审批岗位标签”，不要再混用角色编码。
3. 如果暂时不想新增字段，就把现有 `firstApproverTag/secondApproverTag` 明确约定为“审批岗位标签”，其取值先与身份标签集合保持一致。

也就是说，请假审批建议统一写成：

- 普通员工：`EMPLOYEE -> HR_SPECIALIST -> HR_MANAGER`
- 部门经理：`MANAGER -> HR_MANAGER -> GENERAL_MANAGER`
- HR经理：`HR_MANAGER -> GENERAL_MANAGER`
- 总经理：`GENERAL_MANAGER -> HR_MANAGER`
- 财务专员：如果你希望他在人事请假上仍归普通员工路径，就让他的主身份仍是 `FINANCE_SPECIALIST`，并新增规则：
  - `FINANCE_SPECIALIST -> HR_SPECIALIST -> HR_MANAGER`

不要再依赖“`FINANCE_SPECIALIST` 自动别名成 `EMPLOYEE`”这种隐式兼容作为主规则。

## 5. 关于“HR专员/财务专员既是普通员工又有特殊权限”的处理

这个矛盾可以这样拆开：

### 业务身份上

他们不是普通员工，而是“带专业条线属性的员工”：

- `HR_SPECIALIST`
- `FINANCE_SPECIALIST`

### 审批归类上

他们在某些审批规则里可以“沿用普通员工路径”，但这不表示他们的身份就是 `EMPLOYEE`。

正确做法是：

1. 身份保持 `HR_SPECIALIST` / `FINANCE_SPECIALIST`
2. 若审批链与普通员工一致，就显式配置一条相同规则
3. 不要靠别名自动回退去凑命中

这样你就能同时满足：

- HR 专员具有人事模块特殊权限
- 财务专员具备薪资模块特殊权限
- 他们请假时仍可走“普通员工式”的审批路径

### HR经理的特殊性

HR经理确实是“部门经理的一种更高阶业务身份”，但不要再把它设计成 `MANAGER` 的别名主逻辑。

建议：

1. 主身份固定为 `HR_MANAGER`
2. 如确有需要，可以在身份兼容表里声明它拥有 `MANAGER` 的某些继承能力
3. 但审批规则匹配时，永远先精确匹配 `HR_MANAGER`

## 6. 关键落地原则

### 原则一：精确匹配优先，别名匹配只做兜底

推荐优先级：

1. 申请人身份精确命中
2. 申请人身份继承命中
3. 通配规则命中

不要再让“别名命中”抢在精确业务身份前面。

### 原则二：审批资格校验按审批岗位标签，不按 roleCode 直接相等

当前最大问题就是：

- 规则里是 `HR_SPECIALIST`
- 登录角色里是 `HR`
- 页面却直接做 `expectedTag === roleCode`

正确做法应改为：

1. 先拿当前用户主身份标签
2. 再拿当前用户可承担的审批岗位集合
3. 判断是否包含当前单据的 `pendingApproverTag`

建议引入一张简单映射表或常量映射：

- `HR` 可承担审批岗位：`HR_SPECIALIST`
- `HR_MANAGER` 可承担审批岗位：`HR_MANAGER`
- `GENERAL_MANAGER` 可承担审批岗位：`GENERAL_MANAGER`
- `FINANCE_MANAGER` 可承担审批岗位：`FINANCE_MANAGER`

如果未来你想支持“一人多岗”，也可以自然扩展。

### 原则三：状态显示与状态存储分离

数据库里只存统一状态码，例如：

- `PENDING`
- `APPROVED`
- `REJECTED`
- `CANCELED`

页面再根据当前审批节点渲染：

- 待 HR 专员审批
- 待 HR 经理审批
- 待总经理审批

如果你暂时不想改状态字段值，也至少要做到：

1. 所有页面共用一套状态字典
2. 不再同时存在“待审批”和“待一级审批”两种基础状态定义

## 7. 最小改动版实施方案

这是最适合你当前阶段的方案，不动四层框架，只修正细节。

### 第一阶段：统一口径

1. 明确文档：审批规则中的审批人字段，统一解释为“审批岗位标签”。
2. 统一字典：建立一份中心化常量，维护
   - 身份标签列表
   - 角色列表
   - 审批岗位映射
   - 显示名称
3. 删除页面内散落的 `roleLabelMap`、标签别名和状态字典副本，统一从一个中心配置读取。

### 第二阶段：修正审批资格判断

1. 请假页、首页待办页的 `canApprove` / `pendingLeaves` 判断，全部改成：
   - 权限通过
   - 审批岗位匹配
   - 范围通过
2. 不再使用 `expectedTag === user.roleCode`。
3. 使用“当前用户审批岗位集合”判断是否能审批。

### 第三阶段：补齐显式规则

把依赖隐式别名的规则补全为显式规则：

- `FINANCE_SPECIALIST`
- `HR_SPECIALIST`
- `FINANCE_MANAGER`

凡是审批路径与 `EMPLOYEE` 相同的，也单独配置一条，避免日后再被“兼容逻辑”影响。

### 第四阶段：把审批链计算下沉后端

后端新增统一能力：

1. 创建请假单时，后端根据申请人身份 + 天数计算审批链
2. 审批请假单时，后端校验当前登录人是否具备审批资格
3. 后端写入下一审批节点

前端只负责：

1. 提交申请
2. 发起审批操作
3. 展示后端返回结果

这样才能彻底结束“首页一套、请假页一套”的分裂。

## 8. 推荐的数据口径

### 8.1 用户资料

登录态建议固定返回：

- `roleCode`
- `identityTagCode`
- `approvalAssigneeTags`
- `permissions`
- `moduleScopes`

其中：

- `roleCode` 用于功能权限
- `identityTagCode` 用于业务身份
- `approvalAssigneeTags` 用于审批资格

### 8.2 规则口径

审批规则建议固定区分：

- `applicantTag`
- `approverTag`
- `scope`

如果以后允许三级审批，建议不要继续堆 `thirdApproverTag`，而是改成审批节点列表；但这属于后续演进，不是当前必须动作。

## 9. 你当前系统最适合采用的最终解释口径

你现在最清晰、最容易答辩也最容易继续维护的说法可以是：

1. 身份标签：定义员工在业务中的主身份，例如 HR专员、财务专员、HR经理、部门经理。
2. 角色：定义员工在系统中的功能权限，例如能否管理员工、能否审批薪资、能否配置规则。
3. 模块范围：定义该身份在某模块能看到本人、本部门还是全公司数据。
4. 审批规则：定义某类申请在不同申请人身份和条件下的审批流转。

进一步补一句：

“HR专员和财务专员不是普通员工标签，只是在部分审批流程上可以复用普通员工路径；HR经理也不是普通部门经理标签，而是在人事领域具备更高审批优先级的独立业务身份。”

这句话会把你目前最混乱的点一次讲清。

## 10. 建议优先级

### 必做

1. 统一审批人编码体系，不再混用 `HR` 和 `HR_SPECIALIST`
2. 审批资格判断从 `roleCode` 相等改为“审批岗位标签匹配”
3. 统一状态字典，去掉“待审批 / 待一级审批”双口径
4. 给财务专员、HR专员等特殊岗位补显式审批规则

### 强烈建议

1. 后端接管审批链计算和审批资格校验
2. 文档、SQL、前端常量三处统一口径
3. 页面只展示后端已决议好的审批节点

### 后续演进

1. 审批岗位从单值升级为可配置集合
2. 审批规则从两级固定结构升级为节点列表结构
3. 增加审批审计日志和规则版本号

## 11. 一句话结论

你的四层框架不需要推翻，真正要做的是：

把“身份标签负责业务身份、角色负责功能权限、模块范围负责数据边界、审批规则负责流转节点”这四件事彻底拆开，不再让角色去替代标签，不再让标签去隐式冒充普通员工，也不再让前端页面各自推导审批链。
