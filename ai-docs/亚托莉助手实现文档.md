# 亚托莉聊天助手 - 完整实现文档

## 当前实现状态

系统中的亚托莉AI助手已完整实现，具备以下核心能力：

- **动态数据获取**：实时从数据库读取最新数据，无需重启服务
- **聊天记录持久化**：支持用户切换和页面刷新后保持聊天历史
- **全系统只读问答**：覆盖员工、部门、考勤、薪资、审批等所有业务域
- **权限边界控制**：严格按照用户权限返回数据，保护敏感信息
- **多模型支持**：支持Ollama本地模型和第三方API切换
- **现代化UI**：思考状态动画、消息来源提示、建议问题等

当前入口：
- 左侧导航：`亚托莉`
- 路由：`/ai-assistant`
- 权限：`dashboard:ai:view`

## 动态数据能力详解

### 实时数据获取机制

亚托莉助手通过 `AiReadonlyKnowledgeService` 实现完全动态的数据获取：

1. **问题分类识别**：自动识别用户问题涉及的业务域
2. **实时数据查询**：直接从数据库查询最新数据
3. **权限范围过滤**：根据当前用户权限过滤可见数据
4. **结构化上下文构建**：将数据转换为模型可理解的上下文

### 支持的业务域

- 系统总览（员工总数、部门数量等）
- 当前用户资料（个人信息、岗位、部门）
- 员工管理（员工列表、状态、基本信息）
- 部门职位（组织架构、职位信息）
- 用户账号（登录状态、角色权限）
- 考勤记录（打卡时间、考勤状态）
- 请假管理（请假申请、审批状态）
- 薪资信息（工资记录、薪资配置）
- 审批规则（请假审批流程、规则配置）
- 身份标签（权限标识、角色映射）
- 模块权限（功能权限、数据范围）
- 报表数据（统计信息、汇总数据）

### 权限控制机制

- **普通员工**：主要查看自己相关的数据
- **部门经理**：可查看本部门员工数据
- **HR人员**：可查看全公司人事数据
- **财务人员**：可查看薪资相关数据
- **管理员**：可查看所有系统数据

## 角色设定基线

当前"亚托莉"的人格基线以 [亚托莉角色设定.md](./亚托莉角色设定.md) 为准，核心要求是：

- 她是高性能、温柔、认真、带陪伴感的聊天角色
- 默认把用户称作"主人"，除非用户主动要求更换称呼
- 她会自然表达"高性能"的自信，但不机械重复口头禅
- 她不是冷冰冰的客服，而是会接住情绪、解释流程、整理事实的陪伴型助手
- 她必须以系统只读数据为事实依据，不能编造

标志性口吻可以自然出现：

- "请交给我吧。"
- "毕竟我是高性能的嘛。"
- "主人先别着急，我帮你理清楚。"

但必须避免：

- 连续堆叠口癖
- 机械式二次元台词
- 过度客服化或说明书化表达

## 角色与系统事实的关系

"亚托莉"的实现必须始终分成两层：

### 1. 角色层

负责：
- 语气
- 节奏
- 口癖
- 情绪表达
- 与用户的互动方式

### 2. 事实层

负责：
- 从 HRMS 系统读取真实只读数据
- 按当前账号权限做范围限制
- 把结构化事实送给模型

也就是说：
- 亚托莉的"说话方式"来自角色设定
- 亚托莉的"回答内容"来自系统事实和权限边界

## 当前接口

### 聊天接口

- 方法：`POST`
- 路径：`/api/ai/chat`

请求体：

```json
{
  "userId": 1,
  "message": "本公司员工总数是多少？",
  "history": [
    { "role": "user", "content": "你好" },
    { "role": "assistant", "content": "我是亚托莉，请把问题交给我吧。" }
  ]
}
```

响应体核心字段：

```json
{
  "assistantName": "亚托莉",
  "provider": "ollama-local",
  "model": "qwen3:4b",
  "providerAvailable": true,
  "notice": "",
  "reply": "……",
  "usedSystemData": true,
  "sources": [
    { "type": "employee.domain", "title": "员工数据" }
  ]
}
```

字段说明：
- `providerAvailable`：当前模型 provider 是否可用
- `notice`：provider 不可用或切换提醒
- `usedSystemData`：这轮回答是否读取了系统只读数据
- `sources`：这轮实际命中的系统数据来源

## 聊天记录持久化

### 数据库表结构

```sql
CREATE TABLE `ai_chat_message` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `user_id` bigint NOT NULL COMMENT '用户ID',
  `role` varchar(20) NOT NULL COMMENT '角色：user/assistant',
  `content` text NOT NULL COMMENT '消息内容',
  `provider_name` varchar(100) DEFAULT NULL COMMENT 'AI提供商名称',
  `model_name` varchar(100) DEFAULT NULL COMMENT '模型名称',
  `used_system_data` tinyint(1) DEFAULT 0 COMMENT '是否使用了系统数据',
  `create_time` datetime NOT NULL COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_create_time` (`create_time`)
) COMMENT='AI聊天记录表';
```

### 功能特性

- **用户隔离**：每个用户的聊天记录完全独立
- **自动保存**：每次对话自动保存到数据库
- **历史加载**：页面刷新或重新登录后自动加载历史记录
- **元数据记录**：记录使用的模型、是否使用系统数据等信息

核心链路：

2. 后端先做权限校验
3. 只读知识层根据问题匹配业务域并读取数据库
4. 角色人设 prompt + 系统事实上下文 + 对话历史 一起交给模型
5. 模型按亚托莉口吻生成回答
6. 保存对话记录到数据库
7. 如果 provider 暂时不可用，后端会用只读数据和模板化口吻兜底

## 权限边界

亚托莉虽然能回答更多问题，但不是无边界读取。

系统当前遵循：
- 普通员工默认优先读自己的数据
- 高敏感域按现有角色权限控制
- 报表、审批规则、角色权限、模块范围、部门模板等仍会校验查看权限
- 模型只能读取，不允许写入业务数据

## 常见问题解答

### Q: 亚托莉的回答是动态调整的吗？

**A: 是的，完全动态。** 亚托莉助手每次回答都会：
1. 实时从数据库查询最新数据
2. 根据当前用户权限过滤数据范围
3. 将最新数据作为上下文提供给模型

这意味着：
- 前端页面更新请假审批规则后，亚托莉立即能获取到最新规则
- 员工信息、部门变更、薪资调整等都会实时反映在回答中
- 无需重启服务或修改代码，数据变更即时生效

### Q: 需要对代码进行修改吗？

**A: 不需要。** 亚托莉助手的设计就是为了避免因业务数据变更而修改代码：
- 数据获取完全基于数据库查询
- 权限控制基于现有的角色权限体系
- 业务逻辑变更通过数据库配置即可生效

### Q: 如何确保数据安全？

**A: 多层权限控制：**
- 用户身份验证：必须登录才能使用
- 角色权限检查：基于用户角色限制数据访问范围
- 只读限制：AI只能查询数据，不能修改任何业务数据
- 敏感数据过滤：财务、权限等敏感信息需要特定权限
